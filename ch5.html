一範疇管理是什麼？ ▼

範疇管理（Scope Management）就是回答這個問題：「這個專案到底要做什麼？不做什麼？」

💡 一句話理解範疇管理
把專案的工作邊界畫清楚，確保只做該做的事，不多也不少。

對應您的銀行 AP 經驗：客戶說要做一個「查詢系統」，結果一直加功能變成「交易系統」，這就是範疇失控。範疇管理就是防止這種事發生的機制。

範疇管理分兩個層面：

層面說明
產品範疇
Product Scope		產品或服務本身的特性和功能，以「需求」為衡量基準
專案範疇
Project Scope		為了交付產品，所需完成的工作，以「專案管理計畫」為衡量基準
二範疇管理的時間軸：六個流程全貌 ▼

範疇管理有六個流程，理解這條線，流程順序題就不用死背：

①
規劃範疇管理 Plan Scope Management 規劃

制定「如何定義、確認、控制範疇」的計畫，是後續所有動作的依據

📄 範疇管理計畫 Scope Management Plan
②
蒐集需求 Collect Requirements 規劃

透過各種方法向利害關係人蒐集需求，確認專案要達成什麼目標

📄 需求文件 Requirements Documentation
📄 需求追溯矩陣 Requirements Traceability Matrix
③
定義範疇 Define Scope 規劃

從所有需求中，確認哪些納入本次專案，哪些不納入，畫出邊界

📄 範疇聲明書 Project Scope Statement（在規劃階段產出）
④
建立工作分解結構 Create WBS 規劃

把範疇拆解成較小、可管理的工作包（Work Package），方便估計和執行

📄 WBS ＋ WBS 說明書 WBS Dictionary
⑤
確認範疇 Validate Scope 監控

客戶或利害關係人「正式驗收」交付成果，確認成果符合需求

⚠️ 注意：這是「客戶驗收」，不是品質管控
⑥
控制範疇 Control Scope 監控

監控範疇是否有偏差，管理所有範疇變更，防止範疇潛變

📄 變更請求 Change Request（若有範疇變更需送 CCB）
⚠️ 流程順序常考陷阱：先有什麼才能做什麼？
「定義範疇之前要先做什麼？」→ 先蒐集需求（Collect Requirements），有了需求才能定義範疇。

「建立 WBS 之前要先做什麼？」→ 先定義範疇（Define Scope），有了範疇邊界才能分解工作。

正確順序：規劃 → 蒐集需求 → 定義範疇 → 建立 WBS → 確認範疇 → 控制範疇
三蒐集需求的工具方法（高頻考點） ▼

Ch5 最多題目考「在這個情境下，應該用哪種蒐集需求的方法？」，要記住每種方法的適用情境：

工具方法特徵 / 適用情境
訪談 Interview一對一深入交流。適合需要深入了解特定人的需求時。
焦點團體
Focus Groups		邀請特定族群（使用者、利害關係人）召開座談會，由主持人引導互動式討論。
關鍵字：特定族群、座談、互動式討論
引導 FacilitationPM 促進跨單位人員討論，讓大家達成共識。跨部門工作坊（Workshop）是典型情境。
關鍵字：跨部門、工作坊、達成共識
腦力激盪
Brainstorming		短時間內提出大量想法，不對任何想法做評估。
關鍵字：大量想法、不做評估、越多越好
親和圖
Affinity Diagram		將腦力激盪得到的大量創意，分析共同點與差異，分類整理。
關鍵字：分析腦力激盪的結果、歸類
心智圖
Mind Mapping		視覺化呈現想法之間的關聯，探索更多想法。
關鍵字：分析想法之間的關聯、探索更多
問卷調查
Questionnaire		需要向大量人員蒐集資料時使用。
關鍵字：大量、廣泛、統計分析
雛形法
Prototype		先建立產品模型，交給使用者體驗並取得回饋，再反覆修正。
關鍵字：建立模型、使用者體驗、反覆修正
標竿法
Benchmarking		參考其他組織或業界的最佳實務，作為比較基準。
關鍵字：參考他人、業界範本、比較
💡 最常混淆的三組：焦點團體 vs 引導 vs 腦力激盪
焦點團體：特定族群 + 座談 + 主持人引導 → 重點在「特定族群互動討論」
引導：跨部門 + 工作坊 + 達成共識 → 重點在「讓不同單位的人達成共識」
腦力激盪：大量想法 + 不評估 + 越多越好 → 重點在「產生大量創意」

記憶訣竅：看題目關鍵字，「座談特定族群」→ 焦點團體；「跨部門共識」→ 引導；「大量創意」→ 腦力激盪。
💡 雛形法（Prototype）的特別說明
雛形法是敏捷思維的體現——不是一開始就把需求定死，而是先做出一個可以給使用者體驗的版本，根據回饋再修正，持續迭代。

考題情境：「專案團隊創建產品前，先建立模型交給使用者體驗，取得具體回饋後，再反覆修正模型，持續進行」→ 選 Prototype（雛形法）。
四WBS 工作分解結構（高頻考點） ▼

WBS（Work Breakdown Structure）是把整個專案範疇，拆解成一層一層、越來越小的工作單位。

💡 用銀行 AP 專案來理解 WBS
假設您要開發一個銀行查詢系統：
最上層：「銀行查詢系統開發」
第二層：需求分析 / 系統設計 / 開發 / 測試 / 上線部署
第三層（以「開發」為例）：前端介面 / 後端 API / 資料庫設計
最底層（Work Package）：「設計帳戶查詢 API」→ 可以實際估計時間和成本的最小單位

WBS 是「交付成果導向」的分解，不是活動清單。

WBS 的核心概念

概念說明
工作包
Work Package		WBS 最底層的元素，是可以實際估計時間、成本、指派負責人的最小單位
控制帳戶
Control Account		在 Work Package 上方，匯總成本和進度的管理點，由 PM 監控
WBS 說明書
WBS Dictionary		說明每個 Work Package 的詳細內容，包含範圍、負責人、驗收標準等
團隊成員不確定自己 Work Package 詳細內容時，查 WBS Dictionary
範疇基準
Scope Baseline		由「範疇聲明書 + WBS + WBS Dictionary」三份文件組成
⚠️ 不包含「範疇管理計畫」

建立 WBS 的兩個方向

方向適用時機
由上而下
Top-down		組織有豐富經驗、有 WBS 範本可以參考時，從大到小分解
由下而上
Bottom-up		組織沒有相關經驗，可以邊執行邊向上合併工作交付成果
滾浪式規劃
Rolling Wave Planning		近期的工作詳細規劃，遠期的工作暫時保持高層次，隨著資訊增加再逐步細化
⚠️ WBS 分解的常見考點
「WBS 要分解到多細？」分解到「可以實際估計時間和成本」的程度就夠了，不是越細越好。

考題說「工作應分解為許多細項，下列何者對工作分解的程度是錯誤的？」
→ 錯誤答案是：「分解到能夠實際估計時間及成本」 → 這是正確的分解程度，不是「錯誤」的！
→ 真正錯誤的是：「不能再進行物理性切割」→ WBS 是管理用的分解，不是物理切割。

WBS 能為誰提供最有效的幫助？

WBS 能同時滿足多個角色的需求：

  • 專案團隊：了解自己負責哪些工作
  • PM（專案經理）：系統地估計各工作包的成本和時程
  • 客戶或發起人：了解專案的分解合理性

所以考題問「WBS 能為誰提供最有效幫助？」答案通常是「以上皆是」。

五關鍵文件詳解 ▼
文件用途 / 考試關鍵
需求文件
Requirements Documentation		記錄所有蒐集到的需求內容，以及需求如何滿足專案目標
→ 查「哪些需求確定納入專案範疇」時用這份
需求追溯矩陣
Requirements Traceability Matrix		確認需求與專案目標的連結關係，追蹤每個需求的來源和狀態
→ 查「需求與目標的連結關係」時用這份
→ 定期確認需求仍與目標對齊時也用這份
需求管理計畫
Requirements Management Plan		說明如何蒐集、分析、記錄和管理需求的計畫
包含：需求的屬性、驗收標準、如何排序
⚠️ 不包含「如何蒐集利害關係人的期望」
範疇聲明書
Project Scope Statement		明確定義專案要做什麼、不做什麼，是範疇的邊界定義
→ 在「定義範疇（Define Scope）」流程產出，屬於規劃階段
WBS 說明書
WBS Dictionary		說明每個 Work Package 的詳細內容
→ 團隊成員不清楚自己負責的 Work Package 詳細內容時查這份
範疇基準
Scope Baseline		由「範疇聲明書 + WBS + WBS Dictionary」組成
⚠️ 不包含範疇管理計畫
⚠️ 最容易混淆：需求文件 vs 需求追溯矩陣
需求文件：記錄需求「內容」，以及這些需求如何滿足目標
需求追溯矩陣：追蹤需求與目標的「連結關係」，確保每個需求都和目標對應

考題：「應定期確認專案需求與專案目標的連結關係，請問可以參考哪一份文件？」
→ 答案：需求追溯矩陣（Requirements Traceability Matrix）
六範疇潛變 Scope Creep（高頻考點） ▼

範疇潛變（Scope Creep）是指未經過正式變更流程，範疇就悄悄地擴大了。這是考試中非常高頻的情境題。

💡 用您的工作經驗理解
您在銀行做 AP 開發，客戶說「順便幫我加一個小功能嘛，很簡單的」，然後一個小功能疊一個小功能，最後專案嚴重超出時程和預算——這就是範疇潛變。

PMP 的應對方式：任何需求變更，不管多小，都要走正式變更流程。

遇到範疇潛變，PM 應該怎麼做？

情境正確做法
團隊成員在會議中提議增加計畫外的新功能PM 說明應集中精力完成計畫內工作，且僅限必要的工作（Control Scope）
客戶對交付成果不滿意，原因是雙方對交付清單沒有明確共識應事先「加強管理利害關係人參與（Manage Stakeholder Engagement）」來預防
專案已接近完工，客戶突然提出重大範疇變更要求客戶提出變更詳細說明及申請，走正式變更流程（不能直接拒絕）
一名利害關係人不斷提出變更，影響專案儘早辨識該利害關係人並納入管理，早期蒐集並確認他的需求
⚠️ 「控制範疇」vs「確認範疇」容易混淆
確認範疇（Validate Scope）：客戶「正式驗收」交付成果的動作，重點是「客戶簽字確認」，屬於監控流程。

控制範疇（Control Scope）：監控範疇是否有偏差，管理變更，防止範疇潛變，也屬於監控流程。

考題情境：「PM 正在規劃 WBS，PM 指出團隊應專注完成計畫內的主要工作，且僅限必要工作」→ 這是在做「控制範疇」，不是確認範疇，因為是在防止範疇擴大。
七題目出題模式與陷阱彙整 ▼
📝 題型1：「目前在做哪個流程？」
  • 「PM 安排與客戶會面，確認公司官網的需求範圍」→ 蒐集需求（Collect Requirements）
  • 「確認哪些需求要納入專案範圍」→ 定義範疇（Define Scope）
  • 「把可交付成果分解成較小的單位或較細的工作包」→ 建立 WBS（Create WBS）
  • 「準備交給客戶確認，需進行哪個流程」→ 確認範疇（Validate Scope）
  • 「PM 發現有人在做計畫外的新功能，阻止並說明專注在計畫內工作」→ 控制範疇（Control Scope）
📝 題型2：「應該用哪種需求蒐集方法？」
  • 快速提出大量想法，不做評估 → 腦力激盪
  • 跨部門工作坊，促進討論達成共識 → 引導（Facilitation）
  • 特定使用者族群座談，互動討論 → 焦點團體
  • 大量使用者做滿意度調查 → 問卷調查
  • 先做模型，使用者體驗後反覆修正 → 雛形法（Prototype）
  • 參考其他飯店的建設資料作為範本 → 標竿法（Benchmarking）
  • 腦力激盪後，分析創意的共同點差異加以分類 → 親和圖
📝 題型3：「應該查哪份文件？」
  • 「不確定負責的 Work Package 詳細內容」→ WBS 說明書（WBS Dictionary）
  • 「確認需求與目標的連結關係」→ 需求追溯矩陣
  • 「確認哪些需求確定納入專案範疇」→ 需求文件
  • 「確認專案範疇基準的組成」→ 範疇聲明書 + WBS + WBS Dictionary（三份合稱範疇基準）
⚠️ 最常答錯的題目
Q：「專案結束蒐集需求流程後會得到哪一份文件，除了記錄所有需求的內容，也說明這些需求如何滿足專案目標？」
→ 答案：需求文件（Requirements Documentation），不是需求追溯矩陣。
需求文件記錄需求內容和如何滿足目標；追溯矩陣是追蹤「連結關係」的。

Q：「完成定義範疇（Define Scope）後，必須將相關結果記錄在哪份文件？」
→ 答案：範疇聲明書（Scope Statement），不是 WBS，也不是需求文件。
八核心思維總結 ▼
💡 Ch5 範疇管理的核心思維
  • 範疇管理的本質是「把邊界畫清楚」，要做什麼、不做什麼都要明確
  • 流程有嚴格順序：先規劃 → 蒐集需求 → 定義範疇 → 建立 WBS → 確認 → 控制
  • WBS 是交付成果導向的分解，不是活動清單，分解到「可以估計時間成本」就夠
  • 確認範疇 = 客戶驗收；控制範疇 = 防止範疇擴大，兩個是不同的動作
  • 範疇潛變的預防靠的是早期充分蒐集需求，並走正式變更流程
💡 對應您的實戰經驗
您18年的開發經驗中，「客戶一直改需求」是最熟悉的痛點。PMP 範疇管理給了您一套正式的應對框架：

• 一開始就用需求追溯矩陣確認每個需求和目標的連結 → 讓客戶難以隨意增加沒有目標依據的需求
• 任何新增需求走變更流程，讓客戶了解變更的代價 → 自然減少無謂的範疇潛變

考試記得：看到「客戶提出新功能」，答案通常不是拒絕，而是「走變更流程評估影響」。
