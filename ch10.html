一溝通管理概念與三種溝通技術▼

溝通管理確保「對的資訊，在對的時間，透過對的管道，傳給對的人」。

溝通技術特性適用情境
互動溝通
Interactive		雙向或多向即時交流，效率最高跨國成員需要即時雙向交流；電話、視訊會議、面對面
推式溝通
Push		單向發送，確保對方收到但不保證讀了組織內部公告、發行未來一年的注意事項；Email、公告
拉式溝通
Pull		將資訊放在某處，由接收者自行取用會議結束後整理資料定期提供特定利害關係人；知識庫、入口網站
⚠️ 三種溝通技術判斷
「跨國成員分散各國，需要雙向或多向即時交流」→ 互動溝通
「組織內部發行了未來一年各部門相關注意事項與詳細說明」→ 推式溝通（單向廣播）
「會議後整理資訊，定期透過 Email 提供給特定利害關係人了解」→ 拉式溝通（放著讓對方取用）

合約管理中的溝通應傾向使用：正式書面溝通（Formal Written）

Lesson Learned（經驗學習）最大用途

「作為未來專案參考的歷史記錄」，不是告知團隊現在的規劃，也不是告知行為。

Lesson Learned 最好由以上皆是（PM、Sponsor、Team）共同提供，不只是 PM 一人。

二溝通管理計畫的用途▼
情境應查文件
某部門未被包含在通知範圍，不知道專案延遲狀態溝通管理計畫（確認該利害關係人是否需要通知，計畫是否有瑕疵）
PM 和利害關係人都認為必須向上呈報，要做為分級上呈的依據溝通管理計畫
專案有多個下包商，歸檔文件避免資料外洩溝通管理計畫
成員向資深經理報告進度，資深經理應詢問 PM 以確保正確資訊成員根據溝通管理計畫（RACI）回應應向誰報告
「為了讓專案資訊能在對的時間透過對的管道傳給正確對象，了解利害關係人對資訊的需求與期望」正在使用溝通需求分析（規劃溝通管理時的方法）
💡 遇到「利害關係人沒收到通知」怎麼辦？
PM 依溝通管理計畫通知了所有人，但有一位利害關係人說沒收到。
→ 正確做法：查閱溝通管理計畫，確認該利害關係人是否需要通知，審查計畫是否有瑕疵
→ 不是直接去查 Email，也不是質問功能經理。先確認計畫本身是否正確。
💡 PM 在宴會上聽到有用資訊怎麼辦？
考題（Q8）：PM 在宴會上碰巧聽到一位部門經理談論對專案有用的資訊。
→ 正確做法：在經驗分享文件中記錄宴會中收集來的專案成功資訊
→ 不是立刻確保每個利害關係人都了解溝通計畫，不是評估計畫是否再加強。
三重要文件對照▼
文件用途
溝通管理計畫說明誰、何時、如何、用什麼管道收到哪些資訊；分級上呈依據；下包商文件歸檔依據
議題紀錄
Issue Log		「有人碰到問題，發現大家都有這個議題但沒提出，避免再發生」→ 建立議題日誌定期追蹤
WBSWBS 能有效地與專案團隊、Sponsor、客戶促進溝通（以上皆是）
⚠️ 成員在 Daily Scrum 提出技術問題
Daily Scrum 不是解決技術問題的地方，只討論三個主題：昨天做了什麼、今天做什麼、什麼事情阻礙工作。
→ 遇到技術問題：Daily Scrum 結束後，另外排時間討論此問題。
四題目出題模式與陷阱彙整▼
📝 題型：「用哪種溝通技術？」
  • 即時雙向交流、跨國成員 → 互動溝通
  • 組織公告、單向發送 → 推式溝通
  • 整理後放在某處讓對方自取 → 拉式溝通
  • 合約管理溝通 → 正式書面溝通
📝 題型：「查哪份文件？」
  • 利害關係人沒收到通知 → 溝通管理計畫（確認計畫是否有瑕疵）
  • 問題大家都有但沒人提出，避免再發生 → 建立議題日誌
  • 成員進度報告應向誰報告 → 溝通管理計畫（RACI）
⚠️ 成員發現問題，最好的處理
成員查看狀況報告，發現延誤，最好的做法：安排會議跟 PM 及相關人員共同討論，不是自己在下一份報告中寫上這個問題，不是要求寫在 issue log 中。
