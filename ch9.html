一資源文件全覽 + RACI（高頻考點）▼
文件用途 / 考試關鍵字
資源管理計畫
Resource Management Plan		說明如何估算、獲得、管理及運用資源的計畫，包含會議紀律、報告規則等
→ 財務長參與太多專案無時間提供文件，要協調資源排程 → 查資源管理計畫
責任分派矩陣
RAM / RACI		顯示每個工作包由誰負責執行、誰需要被通知
→ 「有效分配專案成員工作」「兩位成員對某活動由誰執行有不同意見」→ 查 RACI
→ 「向新成員說明角色之間的報告關係」→ 查 RAM
專案組織圖
Project Organization Chart		呈現各部門成員的報告關係，一覽所需人數
→ 「了解各部門成員的報告關係及所需人數」→ 查專案組織圖
資源分解結構
RBS		以階層方式列出所有資源類別，了解資源種類、數量、等級與技術水準
→ 「了解活動所需的資源種類、數量、等級與技術水準」→ 查 RBS
資源行事曆
Resource Calendar		記錄每個資源的可用時間
→ 「考量資源可用時間估算工期」→ 參考資源行事曆
團隊章程
Team Charter		記錄團隊的工作守則、會議規範、溝通規則
→ 「成員常無法準時出席、會議被打斷無法準時結束，避免再發生」→ 查團隊章程
⚠️ 最容易混淆：組織圖 vs RAM vs RBS
組織圖：看「誰向誰報告」、總共需要幾個人
RAM / RACI：看「每個工作包由誰負責、誰執行、誰需被告知」
RBS：看「需要哪些種類的資源、什麼技術水準」

RACI 責任分派矩陣

R
Responsible（負責執行）
實際執行這項工作的人，一個任務可以有多個 R
A
Accountable（當責）
對這項工作最終結果負責的人。每個任務只能有一個 A
C
Consulted（諮詢）
需要被徵詢意見的人（雙向溝通）
I
Informed（告知）
需要被通知結果的人（單向溝通）
⚠️ RACI 最重要的考點
「RACI 矩陣中，僅能有一個人的角色是？」→ Accountable（A）
每個任務只能有一個當責者，R/C/I 都可以多人，只有 A 是唯一的。
二衝突管理：五種解決技巧（超高頻考點）▼
解決問題 Problem Solving
直接面對問題，雙方坐下來找出讓雙方都滿意的解決方案
最持久、最有效 → 首選
妥協 Compromising
雙方各退一步，找到都能接受的中間方案，沒有完全的贏家或輸家
雙方部分滿足
緩和 Smoothing
強調雙方共同點，淡化差異，暫時緩解緊張關係，但問題還在
治標不治本
強迫 Forcing
用權力強制要求一方接受，贏者全拿，效果快但傷關係
負面的持久影響
迴避 Withdrawal
不面對問題，暫時脫離衝突情境，問題完全沒解決
最消極、最無效
💡 考試判斷邏輯
「哪種方法影響可持續最久？」→ 解決問題（正面持久）
「PM 說我現在沒空處理！」→ 迴避
「PM 與 Sponsor 技術上時常有不同意見」→ 解決問題
「兩個成員對測試方法有不同意見」→ 召集兩人討論爭議點（解決問題）

⚠️ 強迫的影響也持久，但是負面的持久；解決問題的影響是正面的持久，要區分清楚。
三團隊發展五個階段（Tuckman 模型）▼
①
組建 Forming
團隊剛成立，成員彼此不熟悉，建立初步認識，想避免未來糾紛
初次見面、辦非正式宴會、初步認識
②
風暴 Storming
對工作方法、技術方向有不同意見，衝突頻繁，開會常不歡而散
意見不合、爭論、無法達成共識
③
規範 Norming
調整彼此工作習慣與行為，逐漸建立共同規範，開始合作
調整習慣、建立規範、開始合作
④
執行 Performing
團隊高度協作，除完成本份工作外也相互解決問題，效率最高
高效協作、互相補位、解決問題
⑤
解散 Adjourning
專案結束，團隊解散，成員各自離開
專案結束、解散
💡 衝突是正常的，不是壞事
風暴期（Storming）是團隊發展的必經過程，PM 的職責是引導衝突往「解決問題」方向走，不是壓制。
記憶口訣：組→風→規→執→散
四馬斯洛需求理論▼
自我實現 Self-Actualization（最高層）實現個人潛能、成長
自尊 Esteem成就感、被尊重、被認可
社會 Social / Love & Belonging歸屬感、人際關係、被接受
安全 Safety工作安全、人身安全、穩定
生理 Physiological（最底層）食物、水、休息、薪資
💡 由下到上：生 → 安 → 社 → 尊 → 現
最常錯的是把「社會」和「自尊」順序搞反，記住社會（Social）在自尊（Esteem）的下面。
💡 考題情境：CPI=1.05，SPI=1.1，但團隊成員抱怨沒有成就感、沒有得到滿意回饋
→ 這是自尊需求沒有被滿足（成就感、被認可）
→ PM 正確做法：審查獎勵制度是否需要改善
→ 不是請客戶快點驗收，不是改善時程表
五矩陣組織的資源問題▼
情境正確做法
稀少的關鍵資源被其他專案共用，要確保資源能被自己使用與功能經理協商
成員被功能經理調去其他專案，PM 缺乏資源與功能經理協商討論
團隊缺乏某技術，且只有這個專案需要向外請求調派有經驗的人力資源（不是安排培訓，因為只有這專案用到）
成員在特定技術上經驗不足，沒有更合適人選安排成員參加培訓
稀少資源只有特定地點才有考慮運用虛擬團隊（Virtual Team）
⚠️ 矩陣組織資源問題：找功能經理
矩陣組織的特性是 PM 和功能經理「共享」資源。當資源被調走或發生爭奪，PM 要找的是功能經理協商，不是 Sponsor，不是其他 PM。

「整合跨領域研究成果及資源，但成員只能兼職參與，哪種組織類型較佳？」
→ 矩陣型（Matrix），既能整合跨領域資源，又允許成員兼職參與。

「矩陣組織中，建設團隊的阻礙是什麼？」
→ 團隊成員是暫時借調的，所以難以激勵
六題目出題模式與陷阱彙整▼
📝 題型1：「這個情境用哪種衝突解決技巧？」
  • 「我現在沒空處理！」→ 迴避（Withdrawal）
  • 「大家各退一步」→ 妥協（Compromising）
  • 「召集兩人討論爭議點，找出解決方案」→ 解決問題（Problem Solving）
  • 「強調大家的共同點，暫時緩和」→ 緩和（Smoothing）
  • 「影響可持續最久的方法」→ 解決問題
📝 題型2：「團隊目前在哪個發展階段？」
  • 辦宴會初步認識、避免未來糾紛 → 組建（Forming）
  • 意見不合、開會不歡而散 → 風暴（Storming）
  • 調整習慣和行為、建立規範 → 規範（Norming）
  • 高效協作、互相幫忙解決問題 → 執行（Performing）
📝 題型3：「應該查哪份文件？」
  • 「有效分配成員工作 / 誰執行誰需被通知」→ RACI / RAM
  • 「各部門報告關係、所需人數」→ 專案組織圖
  • 「資源種類、數量、等級、技術水準」→ RBS
  • 「避免會議被打斷、無法準時出席的問題再發生」→ 團隊章程
  • 「資源被過度使用，協調資源排程」→ 資源管理計畫
💡 Ch9 核心思維
  • 衝突解決首選「解決問題」——最持久、最有效，幾乎是所有衝突題的最佳答案
  • RACI 只有 A 是唯一的——每個任務只能有一個當責者
  • 馬斯洛由下到上：生理→安全→社會→自尊→自我實現，社會在自尊的下面
  • 矩陣組織資源問題找功能經理，不是 Sponsor 或其他 PM
  • 風暴期是正常的，衝突不代表團隊有問題，PM 要引導往解決問題的方向
