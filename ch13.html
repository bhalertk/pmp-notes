一四個流程 + 辨識要素▼
流程核心工作考試關鍵字
辨識利害關係人
Identify Stakeholders		找出所有會影響或受到專案影響的人，分析其利益、參與度、影響力「釐清有哪些利害關係人」「分析其權力、利益、關注」
⚠️ 整個專案生命週期都要辨識，不只是起始階段
規劃利害關係人參與
Plan Stakeholder Engagement		制定讓利害關係人有效參與專案的計畫「說明如何讓利害關係人有效參與」「提高支持及降低抗拒，需推動哪份計畫」→ 利害關係人參與計畫
管理利害關係人參與
Manage Stakeholder Engagement		與利害關係人溝通合作，解決問題，管理期望「另一位 PM 表現良好但你的專案延遲，怎麼辦」→ 與計畫經理討論
監控利害關係人參與
Monitor Stakeholder Engagement		監控利害關係人的參與度，根據需要調整策略持續監控參與度變化

辨識利害關係人的識別要素

  • 權力（Power）
  • 利益（Interest）
  • 關注（Concern）
⚠️ 辨識利害關係人「不包含」什麼？
→ 不包含：需求（Needs）
需求是在蒐集需求（Ch5）的範疇，不是辨識利害關係人時的識別要素。
💡 PM 獲得章程後的第一步
「PM 被指派負責新專案並獲得專案章程，首先做什麼？」
→ 辨識所有的利害關係人
拿到章程代表起始階段，起始階段最重要的兩件事：發展章程 + 辨識利害關係人。

「新加入的利害關係人在最終驗收時才表示產品不符合需求，最可能的原因？」
→ 專案過程中未定期辨識及分析利害關係人
二權力/利益方格 + 顯著性模型▼

Power / Interest Grid（權力/利益方格）

Manage Closely
高權力 + 高利益
密切管理，積極參與
Keep Satisfied
高權力 + 低利益
讓他們滿意，避免失去支持
Keep Informed
低權力 + 高利益
定期告知，讓他們了解進度
Monitor
低權力 + 低利益
監控即可，不需花太多心力

顯著性模型（Salience Model）

用三個維度評估利害關係人的重要性，決定優先管理順序：

維度說明
權力 Power對專案施加影響的能力（決策權、資源控制）
合法性 Legitimacy在專案中的參與是否被視為合適、正當
緊迫性 Urgency對專案的需求是否需要立即關注
💡 三維度組合決定優先級
三個維度都具備（權力+合法性+緊迫性） → 最重要，立即管理
只有兩個維度 → 中等重要，需要關注
只有一個維度 → 較不重要，監控即可

考試情境：「這位利害關係人影響力大、需求合理、而且需要立即回應」→ 三維度都具備，最高優先級

三參與程度五個等級▼
等級說明PM 目標
不知情 Unaware不知道專案存在或對其的影響告知資訊
抵制 Resistant知道專案但反對理解訴求，設法轉化
中立 Neutral知道專案但不支持也不反對爭取支持
支持 Supportive知道專案且支持維持並善用支持
領導 Leading積極推動專案成功讓他發揮影響力
⚠️ 利害關係人是抵制狀態，PM 應如何？
「關鍵利害關係人目前是抵制狀態，但期望他是支持狀態」
→ 先了解他抵制的原因 → 針對性溝通 → 讓他了解專案對他的利益 → 逐步轉化為支持
核心：先了解原因，再針對性溝通，不是強迫接受
四題目出題模式與陷阱彙整▼
📝 題型1：「目前在哪個流程？」
  • 「釐清有哪些利害關係人」→ 辨識利害關係人
  • 「說明如何讓利害關係人有效參與」→ 規劃利害關係人參與
  • 「提高支持、降低抗拒」→ 推動利害關係人參與計畫
📝 題型2：「這個利害關係人用哪種策略？」
  • 高權力 + 高利益 → Manage Closely（密切管理）
  • 高權力 + 低利益 → Keep Satisfied（讓他們滿意）
  • 低權力 + 高利益 → Keep Informed（定期告知）
  • 低權力 + 低利益 → Monitor（監控）
📝 題型3：「遇到這個情況 PM 應怎麼做？」
  • 發現專案延誤，Sponsor 被蒙在鼓裡 → 如實報告給 Sponsor
  • 利害關係人要求增加範疇，Sponsor 不喜歡 → 評估影響，走正式變更流程
  • 客戶指定使用品質不合格的子公司 → 子公司無法改善品質就更換廠商
  • 發現產品缺陷但客戶無法察覺 → 與團隊討論解決方案，向客戶提供建議，不能隱瞞
  • 機密資料被外部機構要求提供 → 聯繫客戶並取得允許後才提供
⚠️ 最常答錯的考點
「與上級主管意見不合，一直難以達成共識」
→ 正確做法：針對爭議點與上級主管討論，不是避免討論，不是退出專案

「重要會議上，另一 PM 的專案一切順利，但你的專案有很多延誤，怎麼做？」
→ 答案：與計畫經理（Program Manager）討論，不是找 Sponsor，這是計畫層級的問題

「辨識利害關係人不包含哪個要素？」→ 需求（Needs），需求屬於 Ch5 範疇管理
💡 Ch13 核心思維
  • 利害關係人辨識貫穿整個生命週期，不只是起始階段
  • 拿到章程的第一步 → 辨識利害關係人
  • 遇到抵制的利害關係人 → 先了解原因再溝通，不是強迫
  • Ch13 和 Ch10 的差異：Ch13 管「人的期望和參與度」，Ch10 管「資訊怎麼傳」
