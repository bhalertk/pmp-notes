一六個流程時間軸▼
①
規劃時程管理 Plan Schedule Management 規劃

制定「如何定義、排序、估算、制定和控制時程」的計畫

📄 時程管理計畫（包含：方法工具、測量準確度、SPI 規則等）
⚠️ 不包含「定義專案成員個人績效的方法」
②
定義活動 Define Activities 規劃

把 WBS 工作包拆解成具體的活動

📄 活動清單（Activity List）= 有哪些活動
📄 活動屬性（Activity Attributes）= 每個活動的詳細說明（描述、依賴、資源、負責人）
📄 里程碑清單（Milestone List）= 重要時間節點，Duration = 0
⚠️ 「了解活動詳細內容與工作」→ 查活動屬性，不是活動清單
③
排序活動 Sequence Activities 規劃

確定活動之間的依賴關係和執行順序，用 PDM 順序圖示法表示

📄 專案時程網路圖（Project Schedule Network Diagram）
④
估算活動工期 Estimate Activity Durations 規劃

考量資源可用時間，估算每個活動需要多少時間完成

由「最熟悉該活動的人」提供估算（不是 PM 自己）
工具：類比估算、參數估算、三點估算（PERT）、由下而上估算
⑤
發展時程 Develop Schedule 規劃

整合所有活動、順序、工期、資源，產出正式時程基準

📄 專案時程（甘特圖 Gantt Chart 最常見）
📄 時程基準 Schedule Baseline
關鍵字：「針對活動開始及完成日達成協議」→ 發展時程
⑥
控制時程 Control Schedule 監控

監控時程執行狀況、管理時程變更

時程落後時啟動壓縮策略（Crashing 或 Fast Tracking）
⚠️ 流程判斷關鍵字
「確定安裝作業的先後順序」→ 排序活動
「規劃完成零件所需進行的活動」→ 定義活動
「考量資源可用時間，計算所有活動持續時間」→ 估算活動工期
「針對活動開始及完成日達成協議」→ 發展時程
「了解活動流程的詳細內容與工作」→ 查活動屬性（Activity Attributes）
二要徑法 CPM + 浮時計算（超高頻考點）▼
概念說明
要徑 Critical Path從開始到結束「最長的路徑」，決定專案最短完工時間
⚠️ 是最長，不是最短！
浮時 Float / Slack活動可以延遲而不影響完工日的時間
要徑上的活動浮時 = 0
多條要徑多條要徑 = 專案風險較高（多條路徑都沒有緩衝空間）
浮時 = 最晚開始時間（LS）－ 最早開始時間（ES）

計算範例

活動表：A=2天、B=7天（依賴A）、C=4天（依賴A）、D=5天（依賴B和C）、E=1天（依賴D）

🔴 要徑：A→B→D→E = 2+7+5+1 = 15天
路徑B：A→C→D→E = 2+4+5+1 = 12天

C 活動的浮時 = 15 − 12 = 3天（C可以延遲3天不影響完工）

B 活動的浮時 = 0（B 在要徑上）

⚠️ 考試最常錯的說法
「要徑是網路圖中最短的路徑」→ 錯！是最長的
「多條要徑使專案更容易管理」→ 錯！反而更難管，風險更高
「PM 自己估算所有活動工期」→ 錯！應由最熟悉活動的人估算
三壓縮時程：Crashing vs Fast Tracking（高頻考點）▼
方法做法代價適用情境
趕工 Crashing投入更多資源（人力、錢）縮短要徑上的活動成本增加，不增加風險有額外人力資源、網路順序不能改
快速跟進 Fast Tracking把原本依序進行的活動改為同時（並行）進行風險增加，可能返工不能加預算、網路邏輯可以調整
💡 判斷用哪種壓縮方法
「網路圖邏輯不能修改，有額外人力」 → Crashing
「不能增加預算，Sponsor 要提前完成」 → Fast Tracking
「要徑和非要徑浮時都是0，資源不足」 → Crashing（唯一選擇），沒有浮時無法平滑，也無法 Fast Tracking
壓縮只對要徑上的活動有效，壓縮非要徑無法縮短工期
💡 PM 發現時程落後，正確做法
→ 評估壓縮時間的多個方案，向管理階層提出建議
→ 不是直接告知無法完工，不是減少產品範疇，不是默默加班
四資源優化：Smoothing vs Leveling▼
技術做法對要徑對完工日
資源平滑 Resource Smoothing只調整非要徑活動的浮時範圍不改變不延後
資源撫平 Resource Leveling連要徑活動都可能調整可能改變通常延後
💡 記憶方法
平滑（Smoothing）= 溫和，只動非要徑浮時 → 不影響完工日
撫平（Leveling）= 激烈，連要徑都動 → 通常會延後完工日

使用順序：先試平滑，平滑後資源仍不足，才用撫平。
要徑活動的資源應優先確保。
⚠️ 「哪種資源優化技術可能改變要徑？」
→ 資源撫平（Resource Leveling）
「資源平滑改變要徑和完工日」→ 錯！平滑不改變要徑。
五估算方法 + 三點估算 PERT▼
方法特點考題關鍵字
類比估算 Analogous參考過去類似專案整體結果，快速但準確度低「向 PMO 索取過去類似專案資料，快速估算整體工期」
參數估算 Parametric單位 × 數量，有公式計算，較準確「從眾多資料中分析每單位人力成本估算」「每坪×坪數」
三點估算 PERT(O+4M+P)÷6，不確定性高時用「資料非常分散」「不確定性高」「多種路線有路況不確定性」
由下而上估算 Bottom-up最詳細準確，從最底層向上加總「最詳細、最準確」
⚠️ 類比 vs 參數最常混淆
類比：「參考過去整體結果」→ 快速但不精確，沒有單位公式
參數：「分析每單位成本再估算」→ 有單位 × 數量的公式計算
關鍵：有沒有「單位×數量」的公式。有就是參數，沒有就是類比。

三點估算 PERT 公式

PERT 期望值 = (O + 4M + P) ÷ 6
O = 最樂觀（Optimistic）｜M = 最可能（Most Likely）｜P = 最悲觀（Pessimistic）

計算範例：O=8天、M=10天、P=12天

PERT = (8 + 4×10 + 12) ÷ 6 = 60 ÷ 6 = 10天

💡 估算資料由誰提供？
→ 最熟悉該活動的人（不是 PM 自己，不是高階主管）
PM 自己估算所有活動是常見錯誤。
六題目出題模式與陷阱彙整▼
📝 題型1：「目前在哪個流程？」
  • 「規劃活動進行的順序」→ 排序活動（Sequence Activities）
  • 「規劃要完成零件所需進行的活動」→ 定義活動（Define Activities）
  • 「考量資源可用時間，計算所有活動持續時間」→ 估算活動工期
  • 「針對活動開始及完成日達成協議」→ 發展時程（Develop Schedule）
📝 題型2：「應該查哪份文件？」
  • 「了解有哪些活動」→ 活動清單（Activity List）
  • 「了解活動流程的詳細內容與工作」→ 活動屬性（Activity Attributes）
  • 「了解重要的交付點、審查點」→ 里程碑清單（Milestone List）
📝 題型3：「要縮短工期，該怎麼做？」
  • 有額外預算和人力，網路順序不能改 → 趕工（Crashing）
  • 不能增加預算，可以調整活動順序 → 快速跟進（Fast Tracking）
  • 要徑和非要徑浮時都是0 → 趕工（唯一選擇）
  • 「減少原本依序進行的活動」→ 快速跟進
📝 題型4：「用哪種估算方法？」
  • 初期快速評估、資料不足 → 類比估算
  • 有公式、單位×數量 → 參數估算
  • 資料分散、不確定性高 → 三點估算（PERT）
  • 最詳細準確 → 由下而上估算
📝 題型5：「資源不足，怎麼處理？」
  • 調整非要徑活動，不影響完工日 → 資源平滑（Smoothing）
  • 連要徑都要動，可能延後完工 → 資源撫平（Leveling）
  • 「哪種技術可能改變要徑？」→ 資源撫平
💡 Ch6 核心思維
  • 要徑 = 最長路徑，要縮短工期只能從要徑下手
  • 有錢有人 → Crashing；不能加預算 → Fast Tracking
  • 平滑不影響完工日；撫平通常會延後
  • 遇到時程落後：評估多個方案 → 向管理層提建議，不是自己默默加班
  • Activity Attributes = 活動詳細說明；Activity List = 活動清單
