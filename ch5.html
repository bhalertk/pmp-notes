一六個流程時間軸▼

範疇管理的本質：把專案邊界畫清楚——要做什麼、不做什麼都要明確。

①
規劃範疇管理 Plan Scope Management 規劃

制定「如何定義、確認、控制範疇」的計畫

📄 範疇管理計畫 Scope Management Plan
📄 需求管理計畫 Requirements Management Plan（說明如何蒐集、分析、記錄和管理需求）
②
蒐集需求 Collect Requirements 規劃

透過各種方法向利害關係人蒐集需求

📄 需求文件（Requirements Documentation）：記錄需求內容，說明需求如何滿足目標
📄 需求追溯矩陣（RTM）：追蹤需求與目標的連結關係，整個生命週期持續追蹤
③
定義範疇 Define Scope 規劃

從所有需求中，確認哪些納入本次專案，哪些不納入，畫出邊界

📄 範疇聲明書 Project Scope Statement
⚠️「完成定義範疇後，記錄結果在哪份文件？」→ 範疇聲明書，不是 WBS
④
建立 WBS Create WBS 規劃

把範疇拆解成較小、可管理的工作包（Work Package）

📄 WBS ＋ WBS Dictionary（說明每個 Work Package 的詳細內容、負責人、驗收標準）
範疇基準 = 範疇聲明書 + WBS + WBS Dictionary（三份，不含範疇管理計畫）
⑤
確認範疇 Validate Scope 監控

客戶或利害關係人「正式驗收」交付成果

⚠️ 這是「客戶驗收簽字」，不是品質管控
順序：先 Control Quality（團隊內部確認）→ 再 Validate Scope（客戶正式驗收）
⑥
控制範疇 Control Scope 監控

監控範疇是否有偏差，管理所有範疇變更，防止範疇潛變

📄 變更請求 Change Request（若有範疇變更需送 CCB）
⚠️ 流程順序常考陷阱
「定義範疇之前要先做什麼？」→ 先蒐集需求
「建立 WBS 之前要先做什麼？」→ 先定義範疇
正確順序：規劃 → 蒐集需求 → 定義範疇 → 建立 WBS → 確認範疇 → 控制範疇
二蒐集需求的工具方法（高頻考點）▼
工具方法特徵 / 適用情境
訪談 Interview一對一深入交流。適合深入了解特定人的需求。
焦點團體 Focus Groups特定族群座談，由主持人引導互動式討論
關鍵字：特定族群、座談、互動式討論
引導 FacilitationPM 促進跨單位人員討論，達成共識。跨部門工作坊（Workshop）是典型。
關鍵字：跨部門、工作坊、達成共識
腦力激盪 Brainstorming短時間內提出大量想法，不做任何評估
關鍵字：大量想法、不做評估
親和圖 Affinity Diagram將腦力激盪的大量創意，分析共同點與差異，分類整理
心智圖 Mind Mapping視覺化呈現想法間的關聯，探索更多想法
問卷調查 Questionnaire需向大量人員蒐集資料
關鍵字：大量、廣泛、統計
雛形法 Prototype先建立產品模型，使用者體驗後反覆修正
關鍵字：建立模型、使用者體驗、反覆修正
標竿法 Benchmarking參考其他組織或業界最佳實務作為比較基準
關鍵字：參考他人、業界範本
💡 最常混淆的三組
焦點團體：特定族群 + 座談 → 重點在「特定族群互動討論」
引導：跨部門 + 工作坊 → 重點在「讓不同單位達成共識」
腦力激盪：大量想法 + 不評估 → 重點在「產生大量創意」

親和圖 vs 心智圖：
親和圖 = 腦力激盪之後，把結果分類整理
心智圖 = 視覺化呈現，探索更多想法和關聯
三WBS + 關鍵文件▼

WBS 核心概念

概念說明
工作包 Work PackageWBS 最底層，可實際估計時間、成本、指派負責人的最小單位
控制帳戶 Control AccountWork Package 上方的匯總管理點，由 PM 監控成本和進度
WBS Dictionary說明每個 Work Package 的詳細內容（範圍、負責人、驗收標準）
→ 成員不清楚自己負責的 Work Package 詳細內容 → 查 WBS Dictionary
範疇基準 Scope Baseline範疇聲明書 + WBS + WBS Dictionary（三份）
⚠️ 不包含範疇管理計畫

WBS 建立方式

方式說明
由上而下 Top-down從大到小分解，有豐富經驗或 WBS 範本可用時
由下而上 Bottom-up先列所有工作項目再向上歸納，沒有相關經驗時
滾浪式規劃 Rolling Wave Planning近期工作詳細規劃，遠期工作暫時保持高層次，隨著資訊增加再逐步細化
⚠️ WBS 分解的考點
WBS 要分解到多細？ → 分解到「可以實際估計時間和成本」就夠，不是越細越好
WBS 是「交付成果導向」的分解，不是活動清單
「WBS 能幫助誰？」→ 通常答案是「以上皆是」（團隊/PM/客戶都受益）

關鍵文件對照

文件用途 / 考試關鍵
需求文件
Requirements Documentation		記錄蒐集到的需求內容，以及需求如何滿足目標
→ 蒐集需求流程的產出；查「哪些需求確定納入專案」
需求追溯矩陣 RTM追蹤需求與目標的「連結關係」，整個生命週期持續追蹤實現狀態
→ 「定期確認需求與目標的連結關係」→ RTM
→ 確認範疇（Validate Scope）時逐一核對需求 → RTM
範疇聲明書
Project Scope Statement		定義專案要做什麼、不做什麼，定義範疇後產出
⚠️ 需求文件 vs RTM 最常混淆
需求文件：記錄需求「內容」是什麼 → 關鍵字「蒐集需求後的產出」「記錄需求內容」
RTM：追蹤需求與目標的「連結關係」→ 關鍵字「確認連結關係」「定期追蹤」「驗收核對」
四範疇潛變 Scope Creep（高頻考點）▼

範疇潛變（Scope Creep）= 未經正式變更流程，範疇悄悄擴大。

情境正確做法
團隊成員在會議提議增加計畫外的新功能PM 說明應專注完成計畫內工作（Control Scope）
客戶對交付成果不滿，原因是對交付清單沒明確共識事先加強管理利害關係人參與，預防期望不對齊
接近完工，客戶突然提出重大範疇變更要求客戶提出變更申請，走正式變更流程（不能直接拒絕）
利害關係人不斷提出變更，影響專案儘早辨識並納入管理，早期蒐集並確認他的需求
⚠️ 控制範疇 vs 確認範疇容易混淆
確認範疇（Validate Scope）：客戶「正式驗收」交付成果，客戶簽字確認，監控流程
控制範疇（Control Scope）：監控範疇是否有偏差，防止範疇潛變，監控流程

考試情境：「PM 指出團隊應專注完成計畫內工作，且僅限必要工作」→ 控制範疇（防止擴大，不是驗收）

Gold Plating（鍍金）= 團隊自己加功能，沒問客戶要不要 → 屬於控制範疇需要防止的行為
Scope Creep（範疇潛變）= 外部需求不斷偷偷增加 → 同樣需要控制範疇防範
五題目出題模式與陷阱彙整▼
📝 題型1：「目前在做哪個流程？」
  • 「PM 安排與客戶會面，確認公司官網的需求範圍」→ 蒐集需求
  • 「確認哪些需求要納入專案範圍」→ 定義範疇
  • 「把可交付成果分解成較小的工作包」→ 建立 WBS
  • 「準備交給客戶確認」→ 確認範疇（Validate Scope）
  • 「PM 發現有人在做計畫外的新功能，阻止並說明專注在計畫內工作」→ 控制範疇（Control Scope）
  • 「確認 WBS 和工作包符合需求，監控範疇有沒有蔓延」→ 控制範疇
📝 題型2：「應該用哪種需求蒐集方法？」
  • 快速提出大量想法，不做評估 → 腦力激盪
  • 跨部門工作坊，促進討論達成共識 → 引導（Facilitation）
  • 特定使用者族群座談，互動討論 → 焦點團體
  • 大量使用者做調查 → 問卷調查
  • 先做模型，使用者體驗後反覆修正 → 雛形法（Prototype）
  • 參考其他飯店的建設資料作為範本 → 標竿法（Benchmarking）
  • 腦力激盪後，分類歸納創意的共同點 → 親和圖
  • 分析需求之間的關聯，探索更多想法 → 心智圖
📝 題型3：「應該查哪份文件？」
  • 「不確定負責的 Work Package 詳細內容」→ WBS Dictionary
  • 「確認需求內容、蒐集需求後的產出文件」→ 需求文件（Requirements Documentation）
  • 「定期確認需求與目標的連結關係」→ 需求追溯矩陣（RTM）
  • 「確認範疇時核對需求是否已實現」→ RTM
  • 「確認專案範疇基準的組成」→ 範疇聲明書 + WBS + WBS Dictionary（三份）
💡 Ch5 核心思維
  • 流程有嚴格順序：蒐集需求 → 定義範疇 → WBS → 確認 → 控制，不能跳步
  • 確認範疇 = 客戶驗收；控制範疇 = 防止範疇擴大，兩個是不同動作
  • Control Quality 先於 Validate Scope：先自己確認品質，再讓客戶驗收
  • 範疇潛變的預防：早期充分蒐集需求，任何變更走正式流程
  • 看到「客戶提出新功能」→ 答案通常不是拒絕，而是「走變更流程評估影響」
