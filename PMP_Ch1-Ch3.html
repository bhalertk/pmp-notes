一什麼是專案？與非專案的區別 ▼

專案 (Project) 的定義：為了創造獨特的產品、服務或結果，所進行的「臨時性」工作。

特性說明
臨時性有明確的開始與結束日期，完成後就解散
獨特性每個專案都不相同，有別於例行重複工作
漸進精化隨著資訊越來越多，計畫會逐步細化
⚠️ 常考陷阱：訂單處理 ≠ 專案
「每天重複處理客戶訂單、解決問題，並確認30天內接受產品」這種重複性、例行性工作，不是專案，而是「重疊進行的工作（Operations）」。

判斷關鍵：有沒有明確終點？每次是否不同？ → 都是「是」才算專案。
二組織架構與專案型態 ▼

考題常考：在不同組織架構下，PM 的權力和資源取得方式有很大差異。

組織類型PM 特徵與考點
職能型
Functional		PM 幾乎沒有權力，資源由功能部門經理控制，PM 只做協調。
矩陣型
Matrix		PM 與功能經理共享資源。
弱矩陣 → PM 較無權；強矩陣 → PM 較有權。
專案型
Projectized		PM 權力最大，資源完全在專案下，專案結束後人員解散或移轉。

📌 專案型組織的四個特徵（常考）

  • 需要向多位主管報告
  • 會同時兼任多個專案
  • 需要對功能經理報告專案績效
  • 專案結束後會解編歸建，或加入其他專案
⚠️ 矩陣型組織的「抱怨」是正常現象
考題中出現「PM 抱怨利害關係人太多、變更造成的問題過多」或「難以讓成員合作執行工作」，這是矩陣型組織的特徵，不代表組織有問題。

矩陣型的優點是：「能整合並協調跨領域的人力資源」。
三PMO 的角色 ▼

PMO 不是做專案的人，而是「管理專案管理」的單位。核心職責：

  • 制定專案管理政策、流程、工具範本與最佳實務標準
  • 確保專案遵守標準、政策與程序
  • 對共用資源進行管理，實施專案稽核
💡 PM vs PMO vs 功能經理 vs Sponsor — 角色分工
角色核心職責
PM負責「實現專案目標、進行專案整合」
PMO制定標準、稽核、管理共用資源
功能經理管理特定部門的人員，不直接管理專案
Sponsor批准商業企劃案和效益管理計畫，授權給 PM
四五大流程群組 ▼

記憶口訣：「起規執監結」

流程群組核心工作 / 常考考點
起始
Initiating		識別利害關係人、發展專案章程（Project Charter）。
注意：利害關係人在整個專案生命週期都可被識別，但最早從起始階段開始。
規劃
Planning		制定詳細計畫。範疇聲明書在此產出。詳細的 PM 計畫需在「專案章程發佈後」才能開始。
執行
Executing		實際執行工作、管理團隊。Kick-off Meeting 在此階段舉行。
監控
Monitoring & Controlling		確認專案產品是否照計畫完成。PM 收到通知後先評估影響，再決定是否變更。
結案
Closing		確認客戶驗收、解散團隊、歸檔文件、進行經驗學習分享（Lessons Learned）。
⚠️ 陷阱：結案「最重要的工作」是什麼？
答案是：確認客戶已正式驗收（Obtain Formal Acceptance），而不是舉辦慶功宴、解散團隊或算成本效益。完成 Lessons Learned 也是重要工作，但「正式驗收」排第一。
💡 快速判斷目前在哪個流程群組
  • 確認章程 / 授權 → 起始
  • 制定計畫 / 確認範疇聲明書 → 規劃
  • 實際執行 / 舉辦 Kick-off → 執行
  • 監控進度 / 評估是否變更 → 監控
  • 驗收 / 歸檔 / 解散 → 結案
五重要文件與用途對照 ▼

這是 Ch1~Ch3 考最多的考點，務必搞清楚每份文件的用途與產生時機：

文件用途 / 考試關鍵
商業企劃案
Business Case		記錄「經濟可行性評估與成本效益分析」。決定是否啟動，並在進行中持續確認是否繼續。由 Sponsor 批准。
效益管理計畫
Benefit Mgmt Plan		記錄專案目標與組織策略目標的連結、預期效益。與 Business Case 都是在「專案章程之前」的輸入文件，由 Sponsor 確認批准。
專案章程
Project Charter		正式授權專案存在，給予 PM 職權。在「起始」階段制定。內容：商業需求、高層次需求、PM 職責與職權。
⚠️ 不含詳細資源估計或詳細風險。
專案管理計畫
Project Mgmt Plan		說明「如何規劃、執行、監控與結束專案」的完整文件。
利害關係人想了解整個計畫 → 給這份。
需在「專案章程發佈後」才能詳細制定。
組織流程資產
OPA		組織過去累積的文件、範本、流程。PM 對新專案沒經驗時，最應先查閱 OPA（包含過去類似專案的記錄）。
變更日誌
Change Log		記錄「已批准的變更請求（Approved Change Request）」的執行狀況。利害關係人問「某個變更目前執行到哪」→ 查 Change Log。
議題紀錄
Issue Log		記錄目前專案中尚未解決的問題。
假設日誌
Assumption Log		記錄專案規劃時的假設條件與限制。
⚠️ 最容易混淆的三組文件
Q：利害關係人想了解「商業可行性」→ Business Case（不是 Charter）

Q：專案章程尚未批准前，想了解專案需求 → Business Case（章程還沒有，只能看商業論證）

Q：想了解 PM 的職權和授權 → Project Charter

Q：想了解整個專案怎麼規劃、執行、監控 → Project Management Plan
六四種管理方法論 ▼

考題會根據「情境描述」讓你選出正確的方法論，關鍵是抓住兩個維度：

方法論判斷條件 / 情境關鍵字
計畫型
Plan-Driven		交付頻率低 + 變動性低 + 目標是「管理成本」。需求穩定、合規要求高。
迭代式
Iterative		交付頻率低 + 變動性低 + 目標是「交付速度」。反覆修正同一產品，每次更完整。
增量式
Incremental		交付頻率高，每次交付一個新的可用功能，像蓋房子一層一層蓋。
敏捷式
Agile		交付頻率高 + 變動性高，快速短期交付重要價值，給客戶回饋持續調整。
例：新冠疫苗研發（動態需求 + 重複活動到正確）也選 Agile。
混合式
Hybrid		利害關係人對 Agile 感興趣，但不願全面採用，需同時結合多種手法。
💡 判斷方法論的兩步驟
Step 1： 看「交付頻率」高或低
Step 2： 看「變動性」高或低 + 目標是成本還是速度

低頻 + 低變 + 成本目標 → 計畫型
低頻 + 低變 + 速度目標 → 迭代式
高頻 → 增量式
高頻 + 高變 → 敏捷式
想用又不敢全用 → 混合式
七Portfolio / Program / Project 的層級 ▼
層級定義
專案
Project		獨特、臨時性的工作，產出特定的產品、服務或結果。
計畫
Program		一群相互關聯的專案，統一管理以獲得個別管理無法達到的效益。
組合
Portfolio		為了「實現組織策略目標」，將專案、計畫及營運組合起來，優先配置資源、使價值最大化的過程。
💡 記憶方式
Portfolio（組合）= 公司整體戰略層級，最大
Program（計畫）= 一群相關專案的集合
Project（專案）= 單一獨特任務，最小
八商業價值 Business Value ▼

商業價值包含「有形與無形的價值」，範圍非常廣：

類型例子
有形價值貨幣、公司股東權益、公共事業與公共利益、固定設施、工具、市場
無形價值商譽、品牌、商標、聲譽
💡 考題提示
題目通常問「何者屬於商業價值的一部份？」，有形 + 無形都算，答案通常是「以上包含有形及無形的價值都屬於商業價值的一部份」。
九題目出題模式與陷阱彙整 ▼
📝 題型 1：「這是哪種組織架構？」
  • 「需要向多位主管報告」→ 矩陣型
  • 「專案結束後解編歸建或加入其他專案」→ 專案型
  • 「功能經理掌控資源，PM 只協調」→ 職能型
📝 題型 2：「目前在哪個流程群組？」
  • 確認章程 / 授權 → 起始
  • 制定計畫 / 確認範疇聲明書 → 規劃
  • 實際執行 / Kick-off → 執行
  • 監控進度 / 評估是否變更 → 監控
  • 驗收 / 歸檔 / 解散 → 結案
📝 題型 3：「這種情況用哪種管理方法？」

對照「四種管理方法論」表格，抓住：頻率高低、變動高低、成本 vs 速度。

📝 題型 4：「應該查哪份文件？」
  • 了解「為什麼要做、值不值得做」→ Business Case
  • 了解「效益和組織策略連結」→ Benefit Management Plan
  • 了解「PM 的職權和授權」→ Project Charter
  • 了解「整個專案怎麼規劃、執行、監控」→ Project Management Plan
  • 找「過去類似專案的記錄與範本」→ OPA（組織流程資產）
  • 查「已批准的變更執行狀況」→ Change Log
📝 題型 5：「PM 沒有相關專案經驗，應該怎麼辦？」

標準答案：先從 OPA（組織流程資產）中調閱類似專案的文件參考，不是直接找人問、不是安排培訓。

⚠️ 高頻陷阱：「PM 自己跳下去做」通常是錯的
PMP 的思維是「按流程、照文件、走正式管道」。遇到問題的標準流程是：
評估影響 → 走變更流程 → 通知相關人員，不是自己直接行動解決。

這對有18年實戰經驗的您來說最需要注意——實戰直覺是「自己搞定」，考題答案是「通知團隊、走程序」。
十核心思維總結 ▼
💡 Ch1~Ch3 的核心思維
  • PMP 的世界裡，一切都要「按流程、照文件、走正式管道」
  • PM 的職責是「整合、協調、實現目標」，不是自己把所有事做完
  • 遇到問題，第一步通常是「評估影響 → 走變更流程 → 通知相關人員」
  • 文件是一切的基礎，不同文件有不同用途，搞清楚「誰批准、誰產出、誰使用」
💡 您的銀行業背景是優勢
銀行工作對「流程合規、稽核、文件管理」有天然的感覺，和 PMP 強調「依流程行事」非常接近。善用這個直覺，但記得考題的世界比較美好——答案永遠比實際工作更理想化 😄
