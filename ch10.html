一三種溝通技術（高頻考點）▼

溝通管理確保「對的資訊，在對的時間，透過對的管道，傳給對的人」。

溝通技術特性適用情境 / 考試關鍵字
互動溝通
Interactive		雙向或多向即時交流，效率最高「跨國成員需要即時雙向交流」「電話、視訊、面對面」
推式溝通
Push		單向發送，確保對方收到但不保證讀了「組織內部發行未來一年注意事項」「Email、公告」
拉式溝通
Pull		將資訊放在某處，由接收者自行取用「會議後整理資料放在知識庫，讓利害關係人自行取閱」「知識庫、入口網站」
⚠️ 三種溝通技術最常考的陷阱
「會議後整理資訊，定期透過 Email 提供給特定利害關係人了解」
→ 這是拉式溝通，不是推式。資訊整理好放在某處讓對方取用 = 拉式。

「組織內部發行了未來一年各部門相關注意事項」→ 推式溝通（單向廣播發出去）

合約管理中的溝通應傾向使用：正式書面溝通（Formal Written）

Lessons Learned（經驗學習）

最大用途：作為未來專案參考的歷史記錄，不是告知團隊現在的規劃。

Lessons Learned 應由PM、Sponsor、Team 共同提供，不只是 PM 一人。

二重要文件與情境判斷▼
文件用途 / 考試情境
溝通管理計畫
Communication Management Plan		 說明誰、何時、如何、用什麼管道收到哪些資訊
→ 「某部門未被通知，不知道專案延遲」→ 查溝通管理計畫，確認計畫是否有瑕疵
→ 「需要分級上呈的依據」→ 溝通管理計畫
→ 「專案有多個下包商，歸檔文件避免資料外洩」→ 溝通管理計畫
→ 「成員進度報告應向誰報告」→ 溝通管理計畫（RACI）
溝通需求分析
Communication Requirements Analysis		 了解利害關係人對資訊的需求與期望，決定用什麼管道、頻率溝通
→ 「為了讓資訊在對的時間透過對的管道傳給正確對象，分析利害關係人需求」→ 溝通需求分析
議題紀錄
Issue Log		 記錄目前尚未解決的問題，定期追蹤
→ 「有人碰到問題，發現大家都有這個議題但沒提出，避免再發生」→ 建立議題日誌
WBS能同時與專案團隊、Sponsor、客戶促進溝通（以上皆是）
💡 利害關係人說沒收到通知，PM 怎麼辦？
→ 查閱溝通管理計畫，確認該利害關係人是否需要通知，審查計畫是否有瑕疵
→ 不是直接去查 Email，不是質問功能經理。先確認計畫本身是否正確。
💡 PM 在宴會上聽到對專案有用的資訊，怎麼辦？
→ 在經驗分享文件（Lessons Learned）中記錄宴會中收集來的資訊
→ 不是立刻通知所有利害關係人，不是評估溝通計畫是否需要加強。
⚠️ Daily Scrum 中有人提出技術問題
Daily Scrum 只討論三件事：昨天做了什麼、今天做什麼、什麼事情阻礙工作。
→ 遇到技術問題：Daily Scrum 結束後，另外排時間討論，不在 Daily Scrum 中解決。
三題目出題模式與陷阱彙整▼
📝 題型1：「用哪種溝通技術（Communication Technology）？」
  • 即時雙向交流、跨國成員 → 互動溝通（Interactive）
  • 組織公告、單向發送出去 → 推式溝通（Push）
  • 整理後放在某處讓對方自行取用 → 拉式溝通（Pull）
  • 合約管理溝通 → 正式書面溝通（Formal Written）
📝 題型2：「查哪份文件？」
  • 利害關係人沒收到通知 → 溝通管理計畫（確認計畫是否有瑕疵）
  • 問題大家都有但沒人提出，避免再發生 → 建立議題日誌（Issue Log）
  • 成員進度報告應向誰報告 → 溝通管理計畫（RACI）
  • 分析利害關係人對資訊的需求與期望 → 溝通需求分析
📝 題型3：「遇到這個情況 PM 應該怎麼做？」
  • 成員查看報告發現延誤 → 安排會議跟 PM 及相關人員共同討論（不是自己寫進報告，不是直接寫 issue log）
  • 利害關係人說沒收到通知 → 查溝通管理計畫，審查計畫是否有瑕疵
  • PM 在非正式場合聽到有用資訊 → 記錄在 Lessons Learned
  • Daily Scrum 有人提出技術問題 → 會後另外安排時間討論
💡 Ch10 核心思維
  • 溝通管理的核心是「對的資訊 → 對的人 → 對的時間 → 對的管道」
  • 遇到溝通問題，先查溝通管理計畫，確認計畫本身是否正確
  • 三種溝通技術靠關鍵字判斷：即時雙向→互動；單向發出→推式；放著讓對方取→拉式
  • 合約相關溝通一律用正式書面
